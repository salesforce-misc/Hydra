= Hydra
:page-layout: home
:page-description: Hydra is a tiny, thread-safe, type-safe library with a fluent DSL for building finite state machines on the JVM — usable from Java and Kotlin. Declare your states and the events that move between them; Hydra makes only the legal moves, models every outcome as a type, and emits an action on each transition.

<section class="hy-band hy-why">
  <header class="hy-band__head">
    <p class="hy-section-marker">// why a state machine</p>
    <h2 class="hy-band__title">Stations, turnstiles, one ride at a time.</h2>
    <p class="hy-band__sub">
      Stations are <strong>states</strong>; the turnstiles between them are <strong>events</strong> — you
      can't teleport between any two stations, only take the connections the map allows. Modelling a
      workflow this way, instead of a tangle of <code>if</code>/<code>else</code> and callbacks, buys three
      things that matter at scale.
    </p>
  </header>
  <ol class="hy-payoffs">
    <li class="hy-payoff">
      <span class="hy-payoff__idx">01</span>
      <h3 class="hy-payoff__title">Legal moves are explicit</h3>
      <p class="hy-payoff__body">Illegal transitions are rejected <em>by construction</em>, not discovered at runtime. The map is the contract.</p>
    </li>
    <li class="hy-payoff">
      <span class="hy-payoff__idx">02</span>
      <h3 class="hy-payoff__title">Failures are types</h3>
      <p class="hy-payoff__body">Every push returns a sealed <a href="transition-outcomes.html"><code>Transition</code></a> — <code>Valid</code>, <code>Invalid</code>, or <code>NoFromState</code>. The caller is forced to handle "this can't happen".</p>
    </li>
    <li class="hy-payoff">
      <span class="hy-payoff__idx">03</span>
      <h3 class="hy-payoff__title">Durable resume</h3>
      <p class="hy-payoff__body">Because the position is an explicit value, a run that dies mid-flight knows exactly which step it was at — and resumes from there.</p>
    </li>
  </ol>
</section>

<section class="hy-band hy-drive">
  <header class="hy-band__head">
    <p class="hy-section-marker">// two ways to drive the line</p>
    <h2 class="hy-band__title">In-memory, or as a pure routing function.</h2>
    <p class="hy-band__sub">
      A plain <code>transition(event)</code> is perfect for a synchronous, single-process machine. But Hydra
      was built for the harder case — where the durable truth lives <em>outside</em> the JVM.
    </p>
  </header>
  <div class="hy-modes">
    <a class="hy-modecard" href="driving-a-machine.html">
      <span class="hy-modecard__tag">synchronous · in-memory</span>
      <h3 class="hy-modecard__title">transition(event)</h3>
      <p class="hy-modecard__body">Match-and-move. Mutates the current state (thread-safe via a single <code>AtomicReference</code>), fires listeners, and hands you the <code>Transition</code>.</p>
      <span class="hy-modecard__go">How to drive it →</span>
    </a>
    <a class="hy-modecard" href="async-orchestration.html">
      <span class="hy-modecard__tag">asynchronous · durable</span>
      <h3 class="hy-modecard__title">readTransitionAndNotifyListeners(…)</h3>
      <p class="hy-modecard__body">Answers "given I finished step X with exit event E, what runs next?" and fires listeners — <strong>without</strong> mutating in-memory state. Your store holds the real position.</p>
      <span class="hy-modecard__go">Drive an orchestrator →</span>
    </a>
  </div>
